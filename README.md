### ALUSel

这个是最复杂的信号，如下是各个指令对应的输出

| type/inst     | opcode | opcode(binary) | ALUSel               |
| ------------- | ------ | -------------- | -------------------- |
| R-type        | 0x33   | 0110011        | 复杂，与fn7和fn3相关 |
| I-type        | 0x13   | 0010011        | 复杂，与fn7与fn3相关 |
| load          | 0x03   | 0000011        | 0(add)               |
| jalr          | 0x67   | 1100111        | 0(add)               |
| csr           | 0x73   | 1110011        | /                    |
| S-Type        | 0x23   | 0100011        | 0(add)               |
| B-Type        | 0x63   | 1100011        | 0(add)               |
| U-Type(auipc) | 0x17   | 0010111        | 0(add)               |
| U-Type(lui)   | 0x37   | 0110111        | 13(bsel)             |
| J-Format      | 0x6f   | 1101111        | 0(add)               |

可以看到，R-Type和I-Type都是与fn7个fn3相关的，这两个每个都需要单独分析(除此之外，只有lui指令为13，其余都为0)

此时如果将opcode和fn7和fn3一样的话，那么最终将成为一个输入超级多的变量，这样是及其复杂的，

但是，将类型分类之后，我们可以分别对R-Type和I-Type进行实现，然后根据指令类型选择

## Test & Debug

custom test都已经保存在了指定的目录下了，这里说几个要注意的事项

1. 在进行load和store指令的测试的时候，避免存在非4的倍数的地址，例如0x0fff，这样可能会造成冲突，而是为0x0ff0
2. load和store指令尽量避免将数据存在0xfff0这样的高地址，这样会与stack冲突，从而造成数据覆盖
3. 这里我将proj2中所有实现的方法(没有涉及读写这样的系统调用的函数)全部都测试了

如何进行调试：

1. 不论是进行Unit test还是integration test时，进行测试后，首先对比给出的student.out和reference.out，找出不同的时间点T
2. 然后找到那个不同时间点对应的指令(由于使用了流水线啊，因此某个时间点的结果是前面的第二个指令执行的结果)

![image-20250814150428160](.image/image-20250814150428160.png)

3. 然后拆分出指令标识的含义(先根据opcode确定指令类型，进行划分，然后根据funct3和funct7来确定指令的类型)

例如这里的ff040413，其二进制表示为1111 1111 0000 0100 0000 0100 0001 0011

- opcode: 0x13，I-type
- funct 3: 000 - addi指令
- rd: x8, rs2: x8
- imm: 0xff0

由此可以确定指令为: `addi x8, x8, 0xff0`

4. 在原先的RISC-V程序中找到对应的汇编指令，从而得知哪一条出现了问题，并且估计当时的寄存器应该是什么值
5. 然后打开对应的电路

![image-20250814150836275](.image/image-20250814150836275.png)

6. 逐个运行该程序(这里可以手动点击CLK，也可以按快捷键Ctrl + T，这样可以在不同的组件界面来执行下一步)
7. 如果程序很大，并且错误的指令在很后面，可以采用的方法
    - 如果是在很大的integration test中，可以缩减程序大小: 例如在在测试向量乘法的时候，有9×9的向量乘法出现了问题，此时进行Debug会有很大的问题，涉及的指令很多，此时可以缩减为1×1，2×2的向量乘法，先解决小规模的问题，然后查看大规模的情况
    - 或者将某个出错的问题单独提取出来，构成Unit test
    - 如果错误的指令在程序很后面，使用Ctrl + T一条一条的过会比较麻烦，可以采用自动模拟

![image-20250814151423599](.image/image-20250814151423599.png)

待程序执行到接近错误指令的时候，停止，然后手动前进

8. 在执行错误指令时，可以逐个排查寄存器的值，立即数的值，以及每个控制信号等，看哪个不符合预期
9. 如果某个寄存器的值出现了问题，或者内存读取的值不符合预取，那么就需要逐个往前排查，看什么时候这个值是出现问题的
